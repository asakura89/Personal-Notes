## User
•	During production on 11th of May 2023, the Application team found some error in the Production Log. One of them was MediaRequestProtection: Missing/Invalid Hash.
•	After doing some investigation and discussing it with Score Architect, MediaRequestProtection basically implemented to prevent one of DDoS attack which is image resizing attack (https://blog.najmanowicz.com/2015/05/15/do-not-turn-sitecore-media-request-protection-off-and-protect-older-sitecore-versions-with-imageguard/ blog of one of the contributors for this function), it was first release in Sitecore 7.5 (https://doc.sitecore.com/xp/en/sdnarchive/SDN5/Products/Sitecore V5/Sitecore CMS 7/ReleaseNotes/ChangeLog/Release History SC75.html). Basically, before version 7.5 Sitecore allowed user to resize the image media by modifying the "h" (height) or "w" (width) parameter (https://www.seanholmesby.com/images-not-resizing-in-sitecore-7-5-sitecore-8-0/), it was vulnerability because attacker can make a script to do hundreds or thousands of request for the media for the same image but different sizes that will make Sitecore generate media cache files which can consumed a lot of CPU time and Disk space, so the question is which one dies first the CPU or the storage.
•	So, basically by implementing this protection, on an incoming media request, Sitecore calculates the hash of the request query string parameters and compares it with the given hash, in this case the original height and width setting (intended height and width) will be included in the encrypted hash. So, if someone tries to change the height or width manually, the hash will be different and trigger this protection and Sitecore will give the attacker the original file instead of what he intended to request, and then log the error.
•	As explain above the media request protection is intended for image resizing attack which mean it should be use for image specifically, but as Sitecore Architect confirmed himself media request also include other file types for example PDF. Because of this there is a known bug that trigger the protection, this is when a Rich Text Editor component is attached with PDF file and then the user clicks the attachment. The Sitecore editor sometimes (We said sometimes because it is not happened in the developer local machine or Development env/SIT environment), adding the "la=en" or "sc_lang=en" paramater which is language parameter. It triggers the hashing evaluation and will find that the hash of the file is not match because of the language parameter is hashed with the query string (confirmed by Sitecore Architect). Sitecore Architect also said that Sitecore team might assess this issue as not a critical one, so they do not patch it even now.
•	About the language parameter, we found one blog assessing this situation that we can completely remove it from protection, so it will not trigger the protection https://mikael.com/2016/11/dealing-with-media-request-protection-errors/#:~:text=If they are not equal%2C Sitecore will just,encountered. The expected hash value%3A ~40 hex digits. But, during our discussion with Sitecore Architect and the production logs we have been given, we found that some error is image error but containing the language parameter. When we ask about this Sitecore Architect said that it is better to keep language parameter protected as the attacker can also request the file in different language than intended which have the same impact to the CPU time and Disk Space.
•	In conclusion, even though the error message of this protection is a little aggressive. It is safe to ignore the message if it contains language parameter and/or other than image extension also, we found that the Sitecore will not flood the logs with the same request or with the same elements, it will log a different element and/or different parameter of the same element. But will always give the attacker the original file and not cache the media requested, so it will not flood the CPU time or Disk Usage. But We need to monitor this kind of error if it has different condition than what mention above.
•	In another note, custom code can also trigger this protection when we are not protecting the URL. When we generate the URL without protection some people encountered that it will not include the needed hash, so the Missing part of Missing/Invalid hash is triggered causing the Sitecore to log it as error. Fortunately, the production logs do not contain any missing hash in all of the errors logged. So, it safe to say that our custom code did not trigger all of these error logs. However, we have encountered them in developer local machine and Development environment, we will try to experiment and try to get rid of them in our developer local machine, the tricky part is that Sitecore is not logging the same element, and even now we cannot consistently trigger this protection (maybe a good thing?). In case anyone wondering how to protect the URL see pict below or visit the discussion https://sitecore.stackexchange.com/questions/249/mediarequestprotection-an-invalid-missing-hash-value-was-encountered.
